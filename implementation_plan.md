# خطة الحماية والتحسين الشاملة لتطبيق CarLocation

تمت دراسة وتحليل جميع ملفات المشروع البرمجية الخاصة بتطبيق الويب (Next.js) وقواعد البيانات (Supabase) بشكل معمق وثاقب. تم رصد مجموعة من الثغرات الأمنية الحرجة (Critical Vulnerabilities)، والأخطاء البرمجية المنطقية (Logical Bugs)، ومشاكل الأداء والتحجيم (Performance & Scaling Issues) التي قد تؤدي إلى انهيار النظام أو تسريب البيانات الحساسة.

فيما يلي تحليل تفصيلي متبوع بخطة عمل محكمة لجعل التطبيق خالٍ تماماً من العيوب ونقاط الضعف.

---

## 1. الثغرات الأمنية المكتشفة (Vulnerability Analysis)

### ⚠️ الثغرة الأولى: ترقية الصلاحيات الذاتية عند التسجيل (Critical: Privilege Escalation via Signup Metadata)
*   **مكان الثغرة**: دالة قاعدة البيانات `public.handle_new_user()` في ملف الميجريشن الأول:
    ```sql
    CREATE OR REPLACE FUNCTION public.handle_new_user()
    RETURNS TRIGGER AS $$
    BEGIN
      INSERT INTO public.profiles (id, email, name, role)
      VALUES (
        NEW.id,
        NEW.email,
        COALESCE(NEW.raw_user_meta_data->>'name', NEW.email),
        COALESCE(NEW.raw_user_meta_data->>'role', 'worker')
      );
      RETURN NEW;
    END;
    $$ LANGUAGE plpgsql SECURITY DEFINER;
    ```
*   **شرح الثغرة**: عند قيام أي مستخدم بالتسجيل عبر واجهة Supabase Auth العامة (حتى لو كانت مغلقة برمجياً في الواجهة، فإن منفذ API مفتوح افتراضياً)، يمكن للمهاجم إرسال حقل `role` مخصص بقيمة `'manager'` في بيانات التسجيل الأولية (`raw_user_meta_data`). تقوم الدالة بقراءة هذه القيمة وتعيين المستخدم كمدير مباشرة، مما يمنحه الصلاحيات الكاملة لقراءة وتعديل وحذف كل شيء.
*   **الحل**: إزالة إمكانية تعيين الدور من الـ Metadata للمستخدم الجديد وتثبيت الدور الافتراضي دائماً كـ `'worker'`. تتم ترقية المستخدم لمدير فقط عبر لوحة تحكم Supabase بشكل يدوي أو عبر إجراء خاص محمي.

### ⚠️ الثغرة الثانية: تعديل الدور الوظيفي ذاتياً (Critical: Role Escalation via Profile Update RLS)
*   **مكان الثغرة**: سياسة الحماية (RLS) لجدول `profiles`:
    ```sql
    CREATE POLICY "Profiles: update own" ON public.profiles FOR UPDATE USING (auth.uid() = id);
    ```
*   **شرح الثغرة**: تسمح السياسة للمستخدم بتحديث السجل الخاص به في جدول `profiles`. وبما أنه لم يتم استبعاد أو تقييد الأعمدة القابلة للتحديث، يمكن لأي عامل (`worker`) إرسال طلب تحديث لبياناته وتعديل حقل `role` إلى `'manager'` بنجاح، متجاوزاً الحماية لأن التحقق مبني فقط على أن `auth.uid() = id`.
*   **الحل**: إضافة Trigger في قاعدة البيانات يمنع نهائياً تعديل العمود `role` إلا إذا كان المستخدم المنفذ للعملية يملك صلاحيات `manager` مسبقاً.

### ⚠️ الثغرة الثالثة: غياب حماية المسارات برمجياً (High: Missing Page-Level Auth Guards)
*   **مكان الثغرة**: صفحات التطبيق في Next.js مثل `/invoices` و `/reports` و `/settings`.
*   **شرح الثغرة**: يتم إخفاء الروابط في القائمة الجانبية (`sidebar.tsx`) فقط باستخدام `managerOnly: true`، لكن لا توجد أي حماية داخل كود الصفحات نفسها لمنع الوصول المباشر عن طريق كتابة الرابط في المتصفح. كما أن برمجية الـ Middleware (`middleware.ts`) تتحقق فقط من وجود جلسة دخول نشطة (`user`) ولكن لا تتحقق من الدور الوظيفي.
*   **الحل**: تطبيق فحص الصلاحية داخل الصفحات الحساسة باستخدام خطاف الصلاحيات `useRole()`، وعرض رسالة "الوصول غير مصرح به" (Access Denied) أو إعادة التوجيه إلى لوحة التحكم الرئيسية في حال كان المستخدم عادياً (`worker`).

### ⚠️ الثغرة الرابعة: تسريب بيانات الفواتير عبر سياسات RLS (High: Invoices Database Leak)
*   **مكان الثغرة**: سياسة القراءة لجدول الفواتير `invoices`:
    ```sql
    CREATE POLICY "Invoices: read all" ON public.invoices FOR SELECT USING (auth.role() = 'authenticated');
    ```
*   **شرح الثغرة**: تسمح هذه السياسة لأي مستخدم مسجل الدخول بقراءة جميع الفواتير في النظام، بما في ذلك الفواتير المالية الحساسة والأرباح وتفاصيل العملاء، بالرغم من أن صفحة الفواتير مخصصة فقط للمدراء.
*   **الحل**: تقييد سياسة قراءة الفواتير لتكون محصورة فقط بالمدراء (`role = 'manager'`).

## 2. العيوب البرمجية والمنطقية (Logical & Architectural Bugs)

### 🔴 الخلل الأول: الفواتير لا يتم إنشاؤها أبداً (Critical: Invoices Never Created)
*   **مكان الخلل**: قاعدة البيانات وتطبيق الويب.
*   **شرح الخلل**: بالرغم من وجود جدول فواتير (`invoices`) وصفحة لعرضها، لا يوجد أي جزء في الكود البرمجي لواجهة الويب (Next.js) أو في دواد وزناد قاعدة البيانات (Triggers) يقوم بإدخال سجل في جدول الفواتير عند إنشاء حجز جديد، أو تحديثه عند الإرجاع أو الإلغاء. وبالتالي يظل جدول الفواتير فارغاً دائماً، وتظهر صفحة الفواتير فارغة لجميع المستخدمين.
*   **الحل**: كتابة Trigger في قاعدة البيانات يعمل تلقائياً عند إدخال أو تعديل جدول الحجوزات (`rentals`) ليقوم بإنشاء الفواتير وتحديث المبالغ وإلغائها تلقائياً وبشكل آمن تماماً (Security Definer) لتفادي مشاكل الصلاحيات RLS للعامل.

### 🔴 الخلل الثاني: خطر انهيار الأداء في صفحة التتبع (High: GPS Tracking Database Spam)
*   **مكان الخلل**: صفحة التتبع `apps/web/src/app/(app)/tracking/page.tsx`
*   **شرح الخلل**: تقوم الصفحة بجلب جميع سجلات التتبع بالكامل من قاعدة البيانات وتصفيتها برمجياً في المتصفح للحصول على آخر موقع لكل سيارة:
    ```typescript
    const { data: trackingData } = await supabase.from("tracking").select("car_id, latitude, longitude, timestamp")...
    ```
    عند تشغيل النظام وتحديث مواقع السيارات باستمرار، سيمتلئ جدول التتبع بملايين السجلات. جلب كل هذه البيانات دفعة واحدة في المتصفح سيؤدي إلى استهلاك هائل للإنترنت وبطء شديد ثم تجمد متصفح المستخدم والـ Database Timeout.
*   **الحل**: إنشاء واجهة عرض (PostgreSQL View) في قاعدة البيانات تعيد فقط السجل الأخير لكل سيارة باستخدام `DISTINCT ON (car_id)` وجلب البيانات من هذه الواجهة مباشرة في واجهة المستخدم، مما يخفض البيانات المسترجعة من ملايين السطور إلى عدد السيارات الفعلي فقط (مثلاً 20 سطر).

### 🔴 الخلل الثالث: استهلاك البيانات ومشاكل التزامن في الإشعارات (Medium: Expiry Checks Performance & Race Conditions)
*   **مكان الخلل**: ملف `apps/web/src/lib/notifications.ts` ودالة `checkExpiryDates()`.
*   **شرح الخلل**: يتم تشغيل فحص تواريخ انتهاء التأمين والفحص التقني وصلاحية رخص القيادة في متصفح العميل عند تحميل صفحة الإشعارات. تقوم هذه الدالة بتحميل كافة السيارات وكافة العملاء من قاعدة البيانات ومعالجتها في المتصفح، ثم حذف وإدخال الإشعارات.
    هذا يسبب:
    1. استهلاكاً غير مبرر للبيانات وسرعة السيرفر مع زيادة أعداد العملاء والسيارات.
    2. مشاكل تزامن (Race Conditions) وتكرار في الإشعارات في حال فتح عدة موظفين التطبيق في نفس الوقت، حيث سيقوم كل متصفح بمحاولة تعديل الإشعارات معاً.
*   **الحل**: نقل هذه العمليات الحسابية بالكامل إلى دالة تخزين (PostgreSQL Function) في قاعدة البيانات ويتم استدعاؤها عبر مهمة مجدولة (Cron Job) يومية، أو استدعاؤها مرة واحدة بأقل عبء.

## 4. خطة التحقق والتمتين (Verification Plan)

### الاختبارات التلقائية وقاعدة البيانات
1.  **اختبار حظر تعديل الأدوار**:
    محاولة تعديل السجل الشخصي لـ `worker` وتغيير الدور لـ `manager` عبر واجهة API والتحقق من رفض قاعدة البيانات للطلب بـ (Trigger Violation).
2.  **اختبار التسجيل الخارجي**:
    محاولة إنشاء مستخدم جديد عبر Auth API ببيانات وصفية (Role Metadata) وتأكيد تسجيله بدور `worker` دائماً.
3.  **اختبار الفواتير التلقائية**:
    إنشاء عقد إيجار جديد، ثم إرجاع السيارة، ثم إلغاء حجز آخر، والتحقق من إنشاء وتعديل الفاتورة بدقة وحساب الغرامات بنسبة 35% بشكل تلقائي.

### التحقق اليدوي في الواجهة
1.  **اختبار حماية الصفحات**:
    تسجيل الدخول بحساب موظف عادي ومحاولة فتح الروابط `/invoices` و `/reports` و `/settings` بشكل مباشر، والتأكد من ظهور صفحة منع الوصول بنجاح.
2.  **اختبار سرعة التتبع**:
    تأكيد تحميل صفحة التتبع فوراً حتى مع وجود آلاف سجلات التتبع في قاعدة البيانات.
